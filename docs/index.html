Architecture

Most monetization plugins are risky because runtime can mutate data. SponsorStream splits responsibilities into a Engine/Studio architecture.

Engine (Runtime)

Campaign matching, read-only retrieval. Called by LLMs.

uv run sponsorstream-engine
  • campaigns_match: Semantic matching with constraints
  • campaigns_explain: Audit trace
  • campaigns_health: Liveness/readiness
  • campaigns_capabilities: Placement + schema info

Studio (Admin)

Provisioning, ingestion, ops. Called by Humans/CI.

uv run sponsorstream-studio
  • collection_ensure: Create collection
  • campaigns_upsert_batch: Ingestion
  • creatives_delete: Cleanup
  • campaigns_report: Analytics summary

Get Started

Prerequisites

Python 3.10+, uv, Docker (for Qdrant)

1. Install uv

curl -LsSf https://astral.sh/uv/install.sh | sh

2. Start Qdrant

docker run -d --name qdrant -p 6333:6333 -p 6334:6334 qdrant/qdrant

3. Sync & Seed

uv sync
uv run sponsorstream-cli create
uv run sponsorstream-cli seed

4. Run Servers

# Engine (LLM-facing)
        uv run sponsorstream-engine

        # Studio (admin)
        uv run sponsorstream-studio

Technical Deep Dive

SponsorStream is built on local embeddings (FastEmbed) and Qdrant. Context embedding stays local; storage and retrieval use your own infrastructure.
